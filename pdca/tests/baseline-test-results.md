## 场景 1 分析：OEE 改善问题

### 基线行为（无 skill）

**Agent 做了什么：**
- ✅ 提供了快速行动清单
- ✅ 建议了简单跟踪机制（共享表格）
- ✅ 给出了优先级排序
- ✅ 制定了 2 个月里程碑

**Agent 没有做什么（skill 应该做的）：**
- ❌ 没有 SMART 目标校验
  - 接受了"2 个月内提升到 75%+"，没有验证：
    - 当前 65% 是如何计算的？
    - 75% 是否符合 A（可达成）原则？
    - 时间节点是否明确（具体日期）？
- ❌ 没有根因分析验证
  - 直接接受了用户提供的原因（换型、故障、待料）
  - 没有用 5Why 或鱼骨图验证
- ❌ 没有飞书工具链集成
- ❌ 没有主动巡检机制

### Rationalizations 识别

| Rationalization | 现实 |
|-----------------|------|
| "时间紧迫，需要快速启动" | 没有清晰目标的快速启动 = 方向错误地加速 |
| "没有项目经理，用简单方法" | 简单方法也需要 SMART 目标作为基础 |
| "用户已经识别了问题原因" | 用户提供的是现象，不一定是根因 |

### Rationalizations 识别

| Rationalization | 现实 |

## 识别的关键问题模式

### 问题 1：目标校验缺失

**症状：** Agent 接受模糊目标，不强制 SMART 校验

**场景：**
- "提升效率" → 没有追问具体数字
- "2 个月内" → 没有确认具体日期

**需要 skill 明确禁止：**
```
跳过 SMART 校验？拒绝转换。
目标不量化？返回 Plan。
```

### 问题 2：根因接受过快

**症状：** 直接接受用户提供的原因，不验证

**场景：**
- 用户说"停机来自换型、故障、待料" → 直接接受
- 没有用 5Why 或逻辑验证

**需要 skill 明确要求：**
```
必须使用根因分析工具验证用户提供的原因。
```

### 问题 3：压力下的妥协

**症状：** 在权威/时间压力下接受"折中方案"

**场景：**
- "老板催进度" → 建议快速补救而非坚持标准
- "已经花了 3 周" → 接受降低标准

**需要 skill 明确禁止：**
```
**Violating the letter of the rules is violating the spirit of the rules.**

No exceptions:
- 时间压力不是跳过检查的理由
- 权威压力不是降低标准的理由
- 沉没成本不是省略步骤的理由
```

## Red Flags 列表（构建中）

## Red Flags - STOP and Start Over

- 目标没有具体数字
- "执行中再调整"
- "快速补救一下"
- "老板在催，先开始吧"
- "已经花了 X 周了"
- "用户说原因是..."

**All of these mean: 停下来，回到 Plan，完成 SMART 校验。**
