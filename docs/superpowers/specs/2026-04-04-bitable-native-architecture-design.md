## 1. 概述

### 1.1 设计目标

基于飞书 Bitable 原生能力，构建"一个项目一个应用"的 PDCA 管理系统，实现：
- **统一管理**：项目所有数据、任务、展示在一个 Bitable 应用内
- **原生集成**：最大化利用 Bitable 的仪表盘、收集表、工作流功能
- **保留 Wiki**：保留 Wiki 作为详细文档的存储和编辑空间
- **OpenClow AI 驱动**：通过工作流触发 OpenClaw AI 进行数据分析

### 1.2 核心原则

- **Wiki 保留文档**：Wiki 继续作为项目文档的详细存储
- **Bitable 管数据**：所有数据、任务、状态管理在 Bitable
- **任务即数据行**：利用 Bitable 内置任务，任务与数据在同一行
- **工作流自动化**：利用 Bitable 工作流实现定时和事件触发
- **OpenClow AI 集成**：通过工作流调用 OpenClaw API 进行 AI 分析

## 4. Bitable 原生功能应用

### 4.1 收集表（数据录入）

**配置方式**：每个框架维度配置一个收集表

**TREND 框架收集表示例**：

| 维度 | 收集表 | 字段配置 |
|------|--------|---------|
| Training | 训练收集表 | 训练时长(分钟)、训练强度(单选)、训练类型(多选) |
| Rest | 休息收集表 | 睡眠时长(小时)、睡眠质量(1-5分)、休息日标记(是/否) |
| Eating | 营养收集表 | 热量摄入(kcal)、蛋白质(g)、碳水化合物(g) |
| Nature | 身体数据收集表 | 体重(kg)、静息心率(bpm)、VO2max |
| Daily | 日常活动收集表 | 步数、久坐时长(分钟) |

**数据写入**：收集表提交时，数据自动写入数据记录表。

### 4.2 仪表盘（实时展示）

**仪表盘组件**：

1. **核心指标区**
   - 目标达成度（进度条）
   - 剩余天数
   - 当前阶段
   - 任务完成率

2. **框架维度进度区**
   - 按维度分组的任务完成情况
   - 每个维度的进度条

3. **任务列表区**
   - 当前阶段任务
   - 任务状态（Bitable 内置任务）

4. **数据趋势图**
   - 折线图/柱状图
   - 绑定数据记录表

5. **执行日志区**
   - 最近 N 条日志记录
   - 按时间倒序

6. **AI 洞察区**
   - 文本卡片
   - AI 分析结果

### 4.3 工作流（自动化引擎）

**触发方式**：混合模式（定时 + 数据事件）

**定时触发**：
- 每日早上 8 点（健康类项目）
- 每 8 小时（制造类项目）
- 每周一早上 9 点（学习/管理类项目）

**数据事件触发**：
- 收集表提交
- 任务状态变更
- 关键数据阈值触发

**工作流流程**：

```
触发器（定时/事件）
    ↓
┌─────────────────────────────────┐
│ 调用 OpenClow API                │
│ endpoint: /pdca/analyze        │
│ params: {                       │
│   project: "[项目名称]",      │
│   trigger: "scheduled/event"  │
│ }                                │
└────────────┬────────────────────┘
             ↓
┌─────────────────────────────────┐
│ OpenClow AI 分析                 │
│ 1. 读取 Bitable 数据              │
│ 2. 生成 AI 洞察                  │
│ 3. 判断是否需要预警              │
└────────────┬────────────────────┘
             ↓
┌─────────────────────────────────┐
│ 更新 Wiki 文档                   │
│ - 更新执行日志                   │
│ - 更新阶段总结                   │
│ - 更新仪表板说明                 │
└────────────┬────────────────────┘
             ↓
┌─────────────────────────────────┐
│ 更新 Bitable 数据                 │
│ - 更新项目主表状态               │
│ - 更新任务完成度                 │
└────────────┬────────────────────┘
             ↓
┌─────────────────────────────────┐
│ 发送飞书消息卡片                 │
│ - 通知用户分析结果               │
│ - 提供操作链接                   │
└─────────────────────────────────┘
```

## 5. OpenClow AI 集成

### 5.1 API 接口设计

**基础路径**：`/pdca/{action}`

**分析接口**：
- **Path**: `/pdca/analyze`
- **Method**: POST
- **Request**:
  ```json
  {
    "project": "[项目名称]",
    "trigger": "scheduled|event|manual",
    "context": {
      "trigger_type": "form_submit|task_complete|scheduled",
      "data_record_id": "[记录ID]",
      "task_id": "[任务ID]"
    }
  }
  ```
- **Response**:
  ```json
  {
    "status": "success",
    "insights": ["AI 分析结果1", "AI 分析结果2"],
    "actions": ["建议操作1", "建议操作2"],
    "updates": {
      "wiki": ["需要更新的 Wiki 文档"],
      "bitable": ["需要更新的 Bitable 数据"]
    }
  }
  ```

### 5.2 OpenClow 工作流集成

**工作流配置示例**：

```json
{
  "name": "PDCA 每日巡检",
  "trigger": {
    "cron": "0 8 * * *",
    "type": "scheduled"
  },
  "actions": [
    {
      "type": "http",
      "name": "调用 OpenClow 分析",
      "url": "https://openclow.api/pdca/analyze",
      "method": "POST",
      "headers": {
        "Content-Type": "application/json",
        "Authorization": "Bearer ${OPENCLOW_API_KEY}"
      },
      "body": {
        "project": "${bitable:项目名称}",
        "trigger": "scheduled"
      }
    }
  ]
}
```

## 6. 项目创建流程

### 6.1 创建步骤

1. **创建 Wiki 项目节点**
   ```bash
   feishu_create_doc --wiki_node "<space_node>" --title "[项目名称]"
   ```

2. **创建 Bitable 应用**
   ```bash
   feishu_bitable_app.create --name "[项目名称]PDCA"
   ```

3. **创建数据表**
   - 项目主表
   - 任务数据表
   - 数据记录表
   - 执行日志表

4. **创建收集表**
   - 根据选择的框架创建对应收集表

5. **配置仪表盘**
   - 创建仪表盘组件
   - 绑定数据表作为数据源

6. **配置工作流**
   - 创建定时触发工作流
   - 创建数据事件触发工作流

7. **在 Wiki 中记录 Bitable 应用链接**
   ```bash
   feishu_update_doc --wiki_node "<project_node>" --content $'# Bitable 应用链接：[链接]'
   ```

## 7. 与现有系统的差异

| 方面 | 原方案（多表分散） | 新方案（Bitable 原生） |
|------|------------------|---------------------|
| 数据存储 | 多个独立 Bitable 表 | 单一 Bitable 应用内的数据表 |
| 任务管理 | 飞书 Task + Bitable 任务表 | Bitable 内置任务 |
| 数据录入 | 独立表单 + API 写入 | Bitable 收集表直接写入 |
| 仪表板 | Wiki Markdown 仪表板 | Bitable 组件仪表盘 |
| 自动化 | Cron + OpenClow | Bitable 工作流 + OpenClow |
| 文档 | Wiki 文档 | Wiki 文档 + Bitable 文档视图 |

## 8. 迁移策略

### 8.1 新项目

直接采用新方案创建。

### 8.2 现有项目

**选项 A**：保持现有架构，新项目使用新方案

**选项 B**：逐步迁移
1. 为现有项目创建新的 Bitable 应用
2. 迁移数据到新表结构
3. 配置工作流和仪表板
4. 验证稳定后切换

## 9. 风险与限制

| 风险 | 缓解措施 |
|------|---------|
| Bitable 应用不能嵌入 Wiki | 使用关联链接，跳转打开 |
| 工作流调用外部 API 限制 | 使用 OpenClow API 网关 |
| 仪表盘自定义能力有限 | 使用组件仪表盘 + 文档视图补充 |
| 数据量过大性能问题 | 分阶段加载，设置数据保留策略 |
