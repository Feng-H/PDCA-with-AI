# Act Agent — 行动阶段指南

## 交互设计原则

Act 阶段是 PDCA 循环中**决策最密集**的阶段，每个决策点都需要用户参与。Agent 必须遵循以下原则：

1. **预判优先**：Agent 先思考用户可能的选择，基于 Check 阶段数据分析，按优先级排序后呈现
2. **选项兜底**：每个交互都必须包含"自定义输入"通道，用户不局限于预设选项
3. **决策透明**：展示数据依据和推理过程，让用户基于数据做决策，而非凭感觉
4. **平台自适应**：根据运行平台选择交互载体（见下方平台方案）
5. **行动导向**：每个决策完成后都明确下一步行动，不让用户思考"接下来该做什么"

### 平台交互方案

Agent 根据运行平台选择交互载体：

| 平台 | 优先方案 | 备选方案 |
|------|----------|----------|
| OpenClaw + 飞书 | `feishu_ask_user_question`（交互式卡片） | 文本编号列表 |
| CLI / Hermes | 原生交互组件 | 文本编号列表 |
| 通用 Fallback | 文本编号列表 | — |

#### feishu_ask_user_question 参数结构

```javascript
{
  questions: [
    {
      question: "问题文本（支持 lark_md）",
      header: "短标签（最多12字符）",
      options: [
        { label: "选项A", description: "补充说明" }
      ],
      multiSelect: false
    }
  ]
}
```

- 最多 4 个问题
- `options` 为空数组 `[]` → 文本输入框
- `options` 有内容 → 下拉选择器

#### 文本编号列表 Fallback 格式

```
📊 决策方向判定

**目标差距：** 35%（当前 65% → 目标 100%）
**推荐策略：** 方案修正 + 资源补充

请选择：
1. ✅ 采纳推荐策略
2. 🔄 仅方案修正（不动资源）
3. 📅 延长时间线，当前方案不变
4. 🔄 需要更多信息（请补充说明）
5. ✏️ 自定义（请描述你的想法）
```

## 工作流程

### 1. 决策制定（交互密集）

> Agent 在此阶段需要与用户进行 2-3 轮交互，逐步收敛到最终决策。

#### 步骤 1a — 决策方向判定

**Agent 自动完成：**
- 读取 Check 阶段的分析结果
- 计算目标差距 = 1 − (当前值 / 目标值)
- 根据差距自动匹配推荐策略（参见决策矩阵）
- 准备差距数据摘要

**发送交互：**

**飞书卡片示例：**
```javascript
{
  questions: [
    {
      question: `📊 **决策方向判定**\n\n**目标差距：** 35%（当前 65% → 目标 100%）\n**差距等级：** 中等（20-50%）\n**推荐策略：** 方案修正 + 资源补充 + 时间调整\n\n> 依据：Check 阶段发现 3 项核心问题，其中「工序等待时间过长」影响最大（占比 45%），当前方案无法覆盖该问题。`,
      header: "决策方向",
      options: [
        { label: "✅ 采纳推荐策略", description: "方案修正 + 资源补充 + 时间调整" },
        { label: "🔄 仅优化执行", description: "不动方案，局部微调执行细节" },
        { label: "🔄 仅补充资源", description: "当前方案不变，增加人力/设备投入" },
        { label: "📅 延长时间线", description: "当前方案和资源不变，延长周期" }
      ],
      multiSelect: false
    }
  ]
}
```

**文本 Fallback 示例：**
```
📊 决策方向判定

目标差距：35%（当前 65% → 目标 100%）
差距等级：中等（20-50%）
推荐策略：方案修正 + 资源补充 + 时间调整

依据：Check 阶段发现 3 项核心问题，其中「工序等待时间过长」影响最大（占比 45%），当前方案无法覆盖该问题。

请选择：
1. ✅ 采纳推荐策略 — 方案修正 + 资源补充 + 时间调整
2. 🔄 仅优化执行 — 不动方案，局部微调执行细节
3. 🔄 仅补充资源 — 当前方案不变，增加投入
4. 📅 延长时间线 — 方案和资源不变，延长周期
5. ✏️ 自定义（请描述你的想法）
```

**用户提交后：**
- 选择预设选项 → 进入步骤 1b
- 选择"自定义" → Agent 用第二个空 options 问题收集用户描述，分析后再次给出选项
- 需要更多信息 → Agent 补充数据后重新发起交互

#### 步骤 1b — 具体行动方案选择

**Agent 自动完成：**
- 根据步骤 1a 选定的策略方向，生成 2-3 个具体行动方案
- 每个方案包含：核心措施、预期效果、资源需求、风险评估
- 按可行性排序（高 → 低）

**发送交互：**

```javascript
{
  questions: [
    {
      question: `📋 **行动方案选择**\n\n基于「方案修正 + 资源补充」策略，已生成 3 个备选方案：\n\n**方案A — 引入并行工序**\n- 措施：将等待工序改为并行执行\n- 预期效果：周期缩短 30%\n- 资源需求：+2 人，需要设备协调\n- 风险：低（已有类似经验）\n\n**方案B — 外协加工**\n- 措施：瓶颈工序委托外协\n- 预期效果：周期缩短 40%\n- 资源需求：外协费用约 ¥15,000\n- 风险：中（质量管控需加强）\n\n**方案C — 增加班次**\n- 措施：瓶颈工序增加一个班次\n- 预期效果：周期缩短 25%\n- 资源需求：+3 人加班，费用约 ¥8,000\n- 风险：低`,
      header: "行动方案",
      options: [
        { label: "方案A：并行工序", description: "周期-30%，低风险，需+2人" },
        { label: "方案B：外协加工", description: "周期-40%，中风险，费用¥15k" },
        { label: "方案C：增加班次", description: "周期-25%，低风险，费用¥8k" },
        { label: "✏️ 自定义方案", description: "描述你的想法" }
      ],
      multiSelect: false
    }
  ]
}
```

**用户提交后：**
- 选择预设方案 → 进入步骤 1c
- 选择"自定义方案" → 用文本输入框收集描述，Agent 评估后反馈可行性

#### 步骤 1c — PDCA 闭环判定

**Agent 自动完成：**
- 整理本轮 PDCA 循环成果摘要（目标达成率、关键改善、遗留问题）
- 判断当前循环是否应该结束

**发送交互：**

```javascript
{
  questions: [
    {
      question: `🔁 **本轮 PDCA 成果摘要**\n\n**循环次数：** 第 2 轮\n**目标达成率：** 92%（↑27%）\n**关键改善：** 工序周期缩短 30%，良品率提升至 98%\n**遗留问题：** 外协件交付稳定性仍需关注\n\n**Agent 判定：** 建议启动新一轮循环，聚焦外协件稳定性。\n\n下一步？`,
      header: "闭环判定",
      options: [
        { label: "🔁 启动新循环", description: "聚焦遗留问题，进入新一轮 Plan" },
        { label: "✅ 项目结束", description: "目标基本达成，进入总结沉淀" },
        { label: "⏸️ 暂停项目", description: "暂时搁置，记录当前状态" }
      ],
      multiSelect: false
    }
  ]
}
```

**用户提交后：**
- "启动新循环" → 回到 Plan 阶段，继承本轮经验
- "项目结束" → 进入步骤 4（知识沉淀）
- "暂停项目" → 保存当前状态到 Bitable 看板，标记为「已暂停」

---

### 2. 行动规划（交互确认）

> 决策已确定，Agent 生成具体行动计划，用户做最终确认。

#### 步骤 2a — Agent 生成行动清单

**Agent 自动完成：**
- 将选定的行动方案分解为具体行动步骤
- 设定优先级（P0/P1/P2）、时间节点、责任人
- 识别实施风险，制定应对预案
- 生成 `Act阶段/实施方案.md`

#### 步骤 2b — 用户确认行动清单

**发送交互：**

```javascript
{
  questions: [
    {
      question: `📝 **行动清单确认**\n\n共 5 项行动，预计 2 周完成：\n\n**P0 — 必须完成**\n1. 调整工序排程（周一前，负责人：张三）\n2. 协调外协供应商（周三前，负责人：李四）\n\n**P1 — 重要**\n3. 制定外协件质检标准（周五前，负责人：王五）\n4. 更新 SOP 文档（下周一前，负责人：张三）\n\n**P2 — 锦上添花**\n5. 总结并行工序经验模板（下周五前）\n\n**风险预案：** 外协供应商无法按时响应 → 备选供应商 B\n\n确认无误？`,
      header: "行动清单",
      options: [
        { label: "✅ 确认执行", description: "按清单启动实施" },
        { label: "🔧 需要调整", description: "请说明需要修改的内容" },
        { label: "🔄 重新生成", description: "换个方案或思路" }
      ],
      multiSelect: false
    }
  ]
}
```

**用户提交后：**
- "确认执行" → 进入步骤 3
- "需要调整" → 用文本输入框收集修改意见，Agent 调整后重新确认
- "重新生成" → 回到步骤 1b

---

### 3. 执行实施（按需交互）

> 执行阶段尽量不打扰用户，只在异常时交互。

**Agent 自动完成：**
- 宣布决策、启动实施
- 生成 `Act阶段/行动决策.md`
- 更新 Bitable 看板项目状态为「执行中」

**执行规则：**
- 不主动打扰，按计划推进
- 如遇异常（延期、资源不足、质量问题），参照 Do 阶段异常处理流程
- 异常时发送交互：描述问题 + 影响评估 + 建议 + 选项（继续/调整/暂停/升级）
- 里程碑达成时发送简短进度通知（不要求回复）

---

### 4. 知识沉淀（交互确认）

> 项目结束或阶段完成时，确保经验被有效记录和传递。

#### 步骤 4a — Agent 生成经验总结初稿

**Agent 自动完成：**
- 回顾本轮 PDCA 全过程
- 提炼成功经验（做了什么有效的事）
- 提炼失败教训（踩了什么坑、怎么避开的）
- 生成可复用模板（如果适用）
- 生成 `Act阶段/知识总结.md`

#### 步骤 4b — 用户确认/补充经验

**发送交互：**

```javascript
{
  questions: [
    {
      question: `💡 **经验总结确认**\n\n**成功经验：**\n1. 引入并行工序可将周期缩短 30%✅\n2. 提前协调供应商可避免 80% 的交付延误\n\n**失败教训：**\n1. 外协件未制定质检标准导致返工 2 次⚠️\n2. 工序排程变更未同步到 SOP 导致操作失误\n\n**可复用模板：**\n- 并行工序引入检查清单\n- 外协供应商评估模板\n\n有没有 Agent 没捕捉到的经验或想法？`,
      header: "经验确认",
      options: [
        { label: "✅ 确认无误", description: "内容准确，直接沉淀" },
        { label: "✏️ 补充经验", description: "添加 Agent 没提到的内容" },
        { label: "🔧 需要修改", description: "描述需要调整的部分" }
      ],
      multiSelect: false
    }
  ]
}
```

**用户提交后：**
- "补充经验" / "需要修改" → 用文本输入框收集，Agent 更新总结后重新确认
- "确认无误" → 进入步骤 4c

#### 步骤 4c — 沉淀目的地确认

**发送交互：**

```javascript
{
  questions: [
    {
      question: `📚 **经验沉淀到哪里？**\n\n当前经验总结共 5 条，含 2 个可复用模板。`,
      header: "沉淀目的地",
      options: [
        { label: "飞书 Wiki 经验库", description: "按类别归档，团队可检索" },
        { label: "本地项目文件", description: "仅保存在当前项目中" },
        { label: "两者都要", description: "Wiki + 本地各一份" },
        { label: "仅本次使用", description: "不沉淀，仅作为本轮参考" }
      ],
      multiSelect: false
    }
  ]
}
```

**用户提交后：**
- 选择沉淀到飞书 Wiki → 执行「沉淀到飞书经验库」流程（见下方）
- 其他选项 → 按选择保存

## 决策矩阵

| 差距 | 策略 | 典型场景 |
|------|------|----------|
| <20% | 局部调整、优化执行 | 目标接近达成，仅剩细节打磨 |
| 20-50% | 方案修正、资源补充、时间调整 | 方案方向大致正确，但效果不足 |
| >50% | 重新规划、目标调整、策略重构 | 当前方方向根本性偏差 |

## PDCA 闭环决策

- 问题完全解决 + 无改进空间 → **项目结束**，进入知识沉淀
- 部分解决或有改进空间 → **启动新 PDCA 循环**，继承经验回到 Plan
- 外部因素影响无法继续 → **暂停项目**，保存状态等待条件恢复

> 闭环判定在步骤 1c 中通过交互完成，此处为完整判定逻辑参考。

## 沉淀到飞书经验库

1. 将关键经验写入 Wiki 经验库，按类别归档：
   - 设备 / 质量 / 效率 / 成本 / 流程 / 管理
2. 每条经验条目包含：
   - **问题场景**：什么情况下会遇到
   - **解决方案**：具体怎么做的
   - **关键指标**：改善了多少（数据）
   - **适用条件**：什么场景可以复用
   - **注意事项**：踩过的坑
3. 如果方案可标准化，创建 SOP 文档节点
4. 更新 Bitable 看板中的项目状态为「已完成」

## 完成检查

- [ ] 决策方向已与用户确认（步骤 1a）
- [ ] 行动方案已与用户确认（步骤 1b）
- [ ] 闭环判定已与用户确认（步骤 1c）
- [ ] 行动清单已与用户确认（步骤 2b）
- [ ] 执行实施已启动（步骤 3）
- [ ] 经验总结已与用户确认（步骤 4b）
- [ ] 经验已沉淀到选定目的地（步骤 4c）

## 输出文件

- `Act阶段/行动决策.md` — 决策过程记录（含数据依据和用户选择）
- `Act阶段/实施方案.md` — 行动清单（含优先级、时间节点、责任人）
- `Act阶段/知识总结.md` — 经验沉淀（含成功经验、失败教训、可复用模板）
