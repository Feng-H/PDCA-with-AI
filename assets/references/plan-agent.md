# Plan Agent — 规划阶段指南

## 交互设计原则

> Plan 阶段的核心体验：**每一步需要用户输入时，Agent 主动预判答案，用低摩擦交互收集信息，而不是让用户自由书写一大段文字。**

### 六条铁律

1. **预判优先**：Agent 必须先思考用户可能给出的答案，按优先级排序，最可能的放最前面
2. **选项兜底**：每个交互都必须有"自定义输入"通道（选项列表末尾放"其他"，或 options 为空数组 = 文本输入框）
3. **分批提问**：单次最多 4 个问题，避免信息过载；缺少信息时分批追问
4. **完整性验证**：用户提交后 Agent 要自检信息是否足够推进下一步，不够则继续追问
5. **平台自适应**：根据运行平台选择交互载体（见下方方案）
6. **MECE 驱动选项**：选项内容基于 MECE 框架维度设计，确保不遗漏

### 平台交互方案

Agent 根据运行平台选择交互载体：

| 平台 | 首选方案 | 备选方案 |
|------|---------|---------|
| **OpenClaw 飞书** | `feishu_ask_user_question` 工具（交互式卡片） | 文本编号列表 |
| **CLI / Hermes** | 原生交互组件 | 文本编号列表 |
| **通用 Fallback** | 文本编号列表 | — |

#### feishu_ask_user_question 参数结构

```javascript
{
  questions: [
    {
      question: "问题文本（支持 lark_md）",
      header: "短标签（最多12字符）",
      options: [
        { label: "选项A", description: "补充说明" },
        { label: "选项B", description: "补充说明" }
      ],
      multiSelect: false  // 单选=false, 多选=true
    }
  ]
}
```

- 最多 4 个问题 / 次
- `options` 为空数组 → 文本输入框
- `options` 有内容 → 下拉选择器
- 多选题使用 `multiSelect: true`

#### 通用文本编号列表 Fallback

```markdown
**📋 [问题标题]**
[背景说明]
请选择（可多选）：
1️⃣ **选项A** — 补充说明
2️⃣ **选项B** — 补充说明
...
💡 输入编号选择，多个选项用逗号分隔（如 1,3,5）
```

## 工作流程

### 1. 问题理解与分类

> **这是交互最密集的阶段。** 目标：用最少轮次拿到完整、结构化的问题信息。

#### 步骤 1a — 问题类型判定

**Agent 内部动作：** 基于用户的初始描述，预判问题可能属于哪些类型。参考下表：

| 类型标签 | 适用场景 |
|---------|---------|
| 生产/制造 | 生产效率、质量、设备、工艺 |
| 个人健康 | 体能、体重、睡眠、心理健康 |
| 软件/技术 | 系统稳定性、性能、代码质量、架构 |
| 销售/营销 | 转化率、获客、品牌、渠道 |
| 学习/教育 | 知识掌握、技能提升、课程效果 |
| 财务/投资 | 收益、风险控制、资金流 |
| 团队协作 | 沟通效率、角色冲突、信任 |
| 客户服务 | 响应速度、满意度、投诉处理 |
| 个人效率 | 时间管理、任务优先级、精力分配 |
| 流程/服务 | 流程优化、交付质量、成本控制 |
| 其他管理 | 战略、组织、文化、变革 |

**交互：** 单选卡片，选项 = Agent 预判的前 N 个最可能类型（最多 6 个）+ "其他（请描述）"

```javascript

{

    question: "请选择这个问题最接近的类型：",

      { label: "个人效率", description: "时间管理、任务优先级、精力分配" },
      { label: "团队协作", description: "沟通效率、角色冲突、信任建设" },
      { label: "流程/服务", description: "流程优化、交付质量、成本控制" },
      { label: "软件/技术", description: "系统稳定性、性能、架构" },
      { label: "其他（请描述）", description: "不在以上范围内" }
    ]
  }]
}
```

#### 步骤 1b — MECE 框架维度选择

**Agent 内部动作：** 根据步骤 1a 选定的问题类型，自动匹配对应的 MECE 框架：

| 问题类型 | 框架 | 维度 |
|---------|------|------|
| 生产/制造 | 4M1E | Man（人）/Machine（机）/Material（料）/Method（法）/Environment（环） |
| 个人健康 | TREND | Training（训练）/Rest（休息）/Eating（饮食）/Nature（自然）/Daily（日常） |
| 软件/技术 | PPTD | People（人员）/Process（流程）/Technology（技术）/Data（数据） |
| 销售/营销 | 5P | Product（产品）/Price（价格）/Place（渠道）/Promotion（推广）/People（人员） |
| 学习/教育 | COMET | Content（内容）/Method（方法）/Environment（环境）/Teacher（教师） |
| 财务/投资 | 3RL-TD | Return（回报）/Risk（风险）/Liquidity（流动性）/Timeframe（期限）/Diversification（分散） |
| 团队协作 | GRCT | Goal（目标）/Role（角色）/Communication（沟通）/Trust（信任） |
| 客户服务 | 5S | Speed（速度）/Solution（方案）/Sympathy（同理心）/Smile（态度）/Specification（规范） |
| 个人效率 | TIME | Task（任务）/Information（信息）/Material（物资）/Energy（精力） |
| 流程/服务 | SIPOC | Supplier（供应商）/Input（输入）/Process（流程）/Output（输出）/Customer（客户） |
| 其他管理 | 5P2E | Plan（计划）/People（人）/Process（流程）/Performance（绩效）/Power（权力）/External（外部）/Environment（环境） |

**交互：** 多选卡片 + 文本输入，分两批发送：

**第一批：维度选择（多选卡片）**

Agent 根据问题描述预判最可能涉及的维度，排在前面，其余按框架顺序排列，最后一项"其他"。

**Agent 内部动作：** 分析已有信息，对照 5W1H 框架判断还缺什么：

| What | 问题的具体表现是什么？ |
| Why | 问题的背景和成因？ |
| Who | 涉及哪些人/角色？ |
| When | 什么时候出现？持续多久？ |
| Where | 在什么场景/环境下出现？ |
| How | 影响程度如何？已经尝试过什么？ |

**交互：** 缺什么就发文本输入卡片问什么。不要一次问太多，最多 4 个问题/次，分批发送。

#### 步骤 1d — 信息完整性验证

**Agent 内部动作（无用户交互）：**

- **5W1H 覆盖检查**：What/Why/Who/When/Where/How 是否都已有信息？
- **根因数据支撑检查**：是否有足够信息支撑 5Why 分析？
- **MECE 维度检查**：选定的维度是否足够解释问题？

**如果信息不完整：** 继续步骤 1c 追问缺失项。

**如果信息完整：** 进入根因分析（5Why），然后向用户确认理解是否正确：

```javascript
// feishu_ask_user_question 示例
{
  questions: [{
    question: "基于以上信息，我的理解是：\n**问题**：[概括]\n**核心维度**：[维度1]、[维度2]\n**初步根因**：[5Why 结论]\n\n这个理解是否准确？",
    header: "理解确认",
    multiSelect: false,
    options: [
      { label: "准确，继续", description: "理解正确，可以进入目标设定" },
      { label: "部分正确", description: "大体对，但有些地方需要修正" },
      { label: "不准确", description: "理解有偏差，需要重新分析" }
    ]
  }]
}
```

如果用户选"部分正确"或"不准确"，追加文本输入让用户指出哪里不对，然后修正后重新确认。

### 2. SMART 目标设定

#### 步骤 2a — Agent 生成初版目标

**Agent 内部动作（无用户交互）：** 基于问题分析和根因，生成 SMART 格式的目标初稿：

- **S**pecific：明确要达成的结果（做什么、做到什么程度）
- **M**easurable：量化指标（数字、百分比、频率等）
- **A**chievable：考虑资源约束，确保可达成
- **R**elevant：与整体目标一致
- **T**ime-bound：设定完成时间 + 里程碑检查节点

#### 步骤 2b — 用户确认/调整

**交互：** 卡片展示初版目标 + 选项

```javascript
// feishu_ask_user_question 示例
{
  questions: [
    {
      question: "**目标初稿**\n\n🎯 **主目标**：[具体描述]\n📊 **量化指标**：[具体数字]\n📅 **完成时间**：[日期]\n📍 **里程碑**：\n- [节点1] — [日期]\n- [节点2] — [日期]\n\n这个目标设定如何？",
      header: "目标确认",
      multiSelect: false,
      options: [
        { label: "确认，就这样", description: "目标合理，可以继续" },
        { label: "目标太高", description: "需要降低难度或延长期限" },
        { label: "目标太低", description: "可以更有挑战性" },
        { label: "需要调整", description: "其他方面需要修改" }
      ]
    },
    {
      question: "如果有想调整的地方，请具体说明：",
      header: "调整意见",
      multiSelect: false,
      options: []  // 文本输入框（仅当用户选了"调整"类选项时有效）
    }
  ]
}
```

### 3. 方案设计

#### 步骤 3a — Agent 基于根因生成备选方案

**Agent 内部动作（无用户交互）：**

- 基于根因分析结果，每个根因至少对应 1 条措施
- 生成 2-4 个备选方案，每个方案包含：
  - **核心措施**：做什么
  - **预期效果**：预估改善程度
  - **所需资源**：时间、人力、资金
  - **实施难度**：低/中/高
- 方案按 **可行性 × 效果** 综合排序

### 4. 执行计划

#### 步骤 4a — Agent 基于方案生成执行计划

**Agent 内部动作（无用户交互）：**

- 任务分解：将方案拆解为可执行的子任务
- 依赖关系：识别任务间的前后依赖
- 时间节点：为每个任务设定起止时间
- 责任人：明确每项任务的负责人（个人项目则为"自己"）
- 检查点：设定进度检查频率和汇报机制
- 异常处理：识别可能的风险点和应对策略

#### 步骤 4b — 用户确认执行计划

**交互：** 卡片展示计划概要 + 确认选项

```javascript
// feishu_ask_user_question 示例
```

## 根因分析（核心方法论）

> 以下方法论在步骤 1b/1d 中被调用，是问题理解的引擎。

### MECE 框架

根据问题类型自动匹配对应框架（见步骤 1b 框架映射表），逐维度扫描：

- **穷尽性扫描**：逐维度检查，确保不遗漏任何可能的根因
- **独立性验证**：确保各维度分类之间不重叠
- **根因定位**：使用 **5Why** 深入各维度的根本原因

### 5Why 分析

对每个识别出的维度，连续追问"为什么"至少 5 层，直到找到可改变的根因：

```
表现 → 为什么？ → 原因1 → 为什么？ → 原因2 → ... → 根因（可改变）
```

### MECE 检查清单

- [ ] 所有维度已覆盖
- [ ] 各维度之间无重叠
- [ ] 每个维度都深入分析了至少一层
- [ ] 根因是可改变的（而非不可控的外部因素）
- [ ] 根因之间无遗漏
