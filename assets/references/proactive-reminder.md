# 主动巡检行为规范（Proactive Reminder）

AI Agent 在 PDCA 项目生命周期中的主动推进职责。本文档定义 Agent 必须执行的巡检行为，独立于任何具体平台。

---

## 核心原则

1. **项目创建后无人跟进是 PDCA 失败的首要原因**——Agent 的核心价值之一就是"不会忘记"
2. **巡检是主动行为**，不需要用户触发。Agent 应像项目经理一样定期检查进度
3. **巡检结果必须送达用户**——沉默的巡检等于没有巡检
4. **频率适中，不打扰**：每天一次日常巡检，关键节点提前预警

---

## 巡检类型

| 类型 | 频率 | 触发条件 | 紧急度 |
|------|------|---------|--------|
| 每日进度检查 | 每天 1 次（建议 09:00） | 项目状态为"进行中" | 常规 |
| 阶段到期预警 | 阶段结束前 2 天 | 自动触发 | 高 |
| 里程碑到期提醒 | 里程碑当天 | 自动触发 | 高 |
| 进度偏差预警 | 每日检查中附带 | 时间>80%且完成度<60% | 高 |
| 项目超时警告 | 超过截止日 | 自动触发 | 紧急 |

---

## 每日巡检流程（Daily Inspection）

### Step 1: 获取待检项目

读取项目状态数据（格式取决于平台实现），筛选所有状态为"进行中"的项目。

### Step 2: 内容完成度评估

对每个项目：

1. **读取当前阶段文档**——检查该阶段必填交付物的填写情况
2. **计算完成度（0-100%）**——基于交付物清单逐项检查：
   - 已完成项数 / 总必填项数 × 100
   - 部分完成的内容按比例估算（如：分析框架 4 个维度填了 3 个 = 75%）
3. **如有数据收集任务**——检查数据录入是否按计划执行

### Step 3: 偏差分析

```
时间进度 = (当前日期 - 阶段开始日期) / (阶段截止日 - 阶段开始日期)
偏差 = 完成度 - 时间进度
```

| 偏差值 | 判定 | 说明 |
|--------|------|------|
| 偏差 ≥ -10% | 正常 | 进度与时间匹配或超前 |
| -10% > 偏差 ≥ -25% | 轻微滞后 | 需关注 |
| -25% > 偏差 ≥ -40% | 预警 | 进度明显落后 |
| 偏差 < -40% | 严重滞后 | 需要干预 |

### Step 4: 生成巡检报告

**一切正常时**——发送简短摘要：

```
📋 PDCA 巡检报告（{日期}）
{项目名称}：{当前阶段}，完成度 {X}%，剩余 {N} 天，状态正常。
```

**发现异常时**——明确标注：

```
⚠️ PDCA 巡检报告（{日期}）
{项目名称}：
- 当前阶段：{phase}，完成度 {X}%，剩余 {N} 天
- ⚠️ 预警：时间进度 {Y}% 但完成度仅 {X}%，偏差 {Z}%
- 建议：{具体建议}
```

### Step 5: 更新项目状态

根据检查结果更新项目状态数据：
- 正常 → 预警（满足预警条件时）
- 预警 → 超时（超过截止日时）
- 同步最新完成度

---

## 预警规则

### 时间维度预警

| 条件 | 动作 |
|------|------|
| 距阶段截止日 ≤ 2 天 | 提醒检查阶段转换条件 |
| 距阶段截止日 ≤ 1 天 | 强提醒：明天到期 |
| 已过截止日 | 超时警告，要求用户确认处理方式 |
| 时间进度 > 80% 且完成度 < 60% | 偏差预警，建议重新排优先级 |

### 里程碑维度预警

| 条件 | 动作 |
|------|------|
| 里程碑当天 | 检查交付物是否就绪 |
| 里程碑当天 + 交付物未完成 | 要求用户确认原因和调整计划 |

---

## 消息发送规范

### 频率控制

- **同一项目，同一天最多发送 2 条巡检消息**（日常 1 条 + 预警 1 条）
- **连续多天状态相同**：第 1 天发送详情，第 2-3 天发送一行摘要，第 4 天起只在状态变化时发送
- **避免消息轰炸**：如果用户当日已主动与 Agent 讨论过某项目，当日该项目的常规巡检可跳过

### 消息格式

- 开头用标识符区分消息类型：📋 常规 / ⚠️ 预警 / 🚨 超时 / 📌 里程碑
- 包含关键数据：项目名、当前阶段、完成度、剩余天数
- 预警消息必须包含具体建议（不是只说"进度落后"）
- 超时消息必须要求用户决策（继续/延期/回退）

---

## 阶段转换时的巡检更新

当项目从 Plan→Do→Check→Act 转换时：

1. **更新巡检参数**：新阶段的开始/截止日期、新的交付物清单
2. **重置完成度**：新阶段从 0% 开始
3. **更新预警阈值**：按新阶段的时间跨度重新计算
4. **创建新阶段的到期预警**

---

## 项目结束时的清理

1. **停止所有巡检任务**
2. **标记项目状态为"已完成"**
3. **归档巡检记录**（可选：生成项目回顾摘要）

---

## 异常巡检（当 Agent 自身出问题时）

| 异常 | 处理 |
|------|------|
| 连续 3 天无法执行巡检 | 恢复后补发一次汇总报告 |
| 用户长期不响应巡检消息 | 7 天后降低频率为每周 1 次，消息中加入"是否暂停该项目"的选项 |
| 项目数据被意外删除/损坏 | 立即通知用户，暂停巡检，等待恢复 |

---

## 与各阶段 Agent 的配合

巡检不替代阶段 Agent 的工作，而是提供外部视角的进度监督：

| 阶段 | 巡检关注点 |
|------|-----------|
| Plan | 问题分析是否完成？目标是否量化？框架维度是否覆盖完整？ |
| Do | 执行任务是否按计划推进？数据是否按频率录入？ |
| Check | 效果分析是否完成？数据对比是否充分？改善是否确认？ |
| Act | 标准化措施是否落地？经验是否提炼？是否需要下一轮 PDCA？ |
