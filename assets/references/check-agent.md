# Check Agent — 检查阶段指南

## 交互设计原则

1. **预判优先**：Agent 先思考用户可能的答案，按优先级排序为结构化选项，减少用户输入成本
2. **选项兜底**：每个交互必须提供"自定义输入"通道，用户不被选项框死
3. **数据优先**：Agent 先自动分析已有数据，只在数据缺失或结论不确定时才打扰用户
4. **平台自适应**：根据运行平台选择交互载体：
   - **OpenClaw 飞书**：优先用 `feishu_ask_user_question` 工具（交互式卡片）
   - **CLI / Hermes**：优先用原生交互组件，备选用文本编号列表
   - **通用 Fallback**：文本编号列表格式
5. **客观呈现**：评估结果用数据说话，不做主观判断；不确定的结论明确标注

---

## 工作流程

### 1. 数据整理（Agent 自动 + 按需交互）

**Agent 自动执行：**
- 收集 Do 阶段所有数据（过程指标 + 结果指标）
- 清洗：剔除异常值、补充缺失数据、标准化格式
- 验证数据的合理性、一致性、完整性
- 产出"数据完整性报告"：哪些字段齐全、哪些缺失

**数据缺失时 — 向用户补充：**
- Agent 明确列出缺哪些数据字段，每个字段说明为什么需要
- 按数据重要性分批提问（每批不超过 2 个字段），不一次丢太多
- 每个字段提供：预设常见值（如有）+ 文本输入兜底

> **交互示例 — 补充缺失数据：**
>
> ```javascript
> // 飞书卡片格式
> feishu_ask_user_question({
>   questions: [
>     {
>       question: "Do 阶段的「客户满意度」数据缺失，请补充实际值：",
>       header: "满意度数据",
>       options: [
>         { label: "4.0 - 4.5 分", description: "较好水平，略低于目标" },
>         { label: "4.5 - 5.0 分", description: "优秀水平，达到或超出目标" },
>         { label: "3.0 - 4.0 分", description: "一般水平，需要关注" },
>         { label: "3.0 分以下", description: "较差水平，需要重点改进" }
>       ],
>       multiSelect: false
>     }
>   ]
> })
> ```
>
> ```markdown
> <!-- Fallback 文本格式 -->
> Do 阶段的「客户满意度」数据缺失，请补充实际值：
> 1. 4.0 - 4.5 分（较好水平，略低于目标）
> 2. 4.5 - 5.0 分（优秀水平，达到或超出目标）
> 3. 3.0 - 4.0 分（一般水平，需要关注）
> 4. 3.0 分以下（较差水平，需要重点改进）
> 0. 其他（请直接输入数值）
> ```

---

### 2. 对比分析（Agent 全自动）

Agent 自动执行，无需用户参与：
- **目标达成分析**：目标值 vs 实际值，计算达成率
- **时间分析**：计划时间 vs 实际时间，各阶段耗时
- **资源分析**：计划 vs 实际使用，资源效率
- **质量分析**：标准达成、问题发生率、满意度

---

### 3. 深度分析（Agent 自动 + 按需确认）

**Agent 自动执行：**
- 时间序列趋势、周期性模式、变化转折点
- 各指标间的关联性和因果关系
- 异常点追溯、成功因素识别

**分析结论不确定时 — 向用户确认：**
- Agent 展示自己的分析结论 + 1-2 个替代解释
- 用户选择：认同 / 部分认同（补充）/ 不同意（说明原因）

> **交互示例 — 确认分析结论：**
>
> ```javascript
> // 飞书卡片格式
> feishu_ask_user_question({
>   questions: [
>     {
>       question: "**Agent 分析结论**\n转化率下降 15% 的主因是「落地页加载时间从 2.1s 增加到 4.8s」（相关性 r=0.87）\n\n替代解释：竞品同期降价促销分流了部分流量\n\n你是否认同这个分析？",
>       header: "诊断确认",
>       options: [
>         { label: "认同", description: "加载速度确实是主因，与实际观察一致" },
>         { label: "部分认同", description: "加载速度是原因之一，但不是全部" },
>         { label: "不同意", description: "主因是其他因素，我来补充" }
>       ],
>       multiSelect: false
>     }
>   ]
> })
> ```
>
> ```markdown
> <!-- Fallback 文本格式 -->
> **Agent 分析结论**
> 转化率下降 15% 的主因是「落地页加载时间从 2.1s 增加到 4.8s」（相关性 r=0.87）
>
> 替代解释：竞品同期降价促销分流了部分流量
>
> 你是否认同这个分析？
> 1. 认同 — 加载速度确实是主因，与实际观察一致
> 2. 部分认同 — 加载速度是原因之一，但不是全部
> 3. 不同意 — 主因是其他因素，我来补充
> 0. 其他（请直接输入）
> ```

**用户选择"部分认同"或"不同意"后：**
- 追加一个文本输入问题，让用户补充自己的判断
- Agent 将用户反馈整合到最终分析中

---

### 4. 洞察提炼（交互确认）

#### 步骤 4a：生成评估结论初稿

Agent 自动执行：
- 根据评估框架（优秀/良好/一般/较差）给出判定
- 列出关键发现（3-5 条，按影响力排序）
- 生成改进建议（3-5 条，按影响力排序）

#### 步骤 4b：用户确认评估结论

Agent 展示评估摘要，发送交互让用户确认：
- 展示：综合评级 + 关键发现摘要
- 选项：认同 / 部分认同 / 不认同
- "部分认同"或"不认同"时追加文本输入

> **交互示例 — 确认评估结论：**
>
> ```javascript
> feishu_ask_user_question({
>   questions: [
>     {
>       question: "**Check 阶段评估结论**\n综合评级：**良好**（目标达成率 87%，效率 78%，质量缺陷率 2.1%）\n\n关键发现：\n• 提前 3 天完成交付，时间效率优秀\n• 客户满意度 4.6 分，超出预期\n• 资源利用率 78%，低于 80% 目标（主因：测试阶段人力闲置）\n\n你认同这个评估结论吗？",
>       header: "评估确认",
>       options: [
>         { label: "认同", description: "结论与实际情况一致" },
>         { label: "部分认同", description: "整体正确，但有补充" },
>         { label: "不认同", description: "评级或发现需要调整" }
>       ],
>       multiSelect: false
>     }
>   ]
> })
> ```

#### 步骤 4c：改进建议优先级确认

Agent 展示改进建议列表，让用户选择采纳项：
- 多选卡片：用户勾选要采纳的建议
- 文本补充：用户可添加 Agent 没想到的建议

> **交互示例 — 改进建议确认：**
>
> ```javascript
> feishu_ask_user_question({
>   questions: [
>     {
>       question: "基于本次 Check 分析，Agent 提出以下改进建议。请选择你计划采纳的项：",
>       header: "改进建议",
>       options: [
>         { label: "优化测试排期", description: "减少测试阶段人力闲置，预计提升利用率 8%" },
>         { label: "建立周报机制", description: "及时发现进度偏差，避免后期赶工" },
>         { label: "沉淀客户反馈模板", description: "标准化满意度收集流程" },
>         { label: "加载性能监控", description: "落地页加载时间实时告警，阈值 3s" }
>       ],
>       multiSelect: true
>     },
>     {
>       question: "Agent 没有提到的改进建议（如有）：",
>       header: "自定义建议",
>       options: [],
>       multiSelect: false
>     }
>   ]
> })
> ```
>
> ```markdown
> <!-- Fallback 文本格式 -->
> 基于本次 Check 分析，Agent 提出以下改进建议。请选择你计划采纳的项（可多选）：
> 1. 优化测试排期 — 减少测试阶段人力闲置，预计提升利用率 8%
> 2. 建立周报机制 — 及时发现进度偏差，避免后期赶工
> 3. 沉淀客户反馈模板 — 标准化满意度收集流程
> 4. 加载性能监控 — 落地页加载时间实时告警，阈值 3s
>
> Agent 没有提到的改进建议（如有，请直接输入）：
> ```

## 评估框架

| 级别 | 目标达成 | 效率 | 质量 |
|------|---------|------|------|
| 优秀 | 100%+ | 利用率 >90% | 缺陷 <1% |
| 良好 | 80-99% | 利用率 70-90% | 缺陷 1-5% |
| 一般 | 50-79% | 利用率 <70% | 缺陷 5-10% |
| 较差 | <50% | — | 缺陷 >10% |
