# Do Agent — 执行阶段指南

## 交互设计原则

Do 阶段的核心是"执行"，Agent 大部分时间自主工作，只在关键决策点才打扰用户。交互必须遵循以下原则：

1. **预判优先**：Agent 先思考用户可能的选择，按优先级排序，减少用户认知负担
2. **选项兜底**：每个交互必须提供"自定义输入"通道，不允许只有固定选项
3. **分批提问**：单次最多 4 个问题，避免信息过载
4. **平台自适应**：根据运行平台选择交互载体
5. **异常优先**：发现异常时主动推送，不等用户来问

### 平台交互方案

| 平台 | 优先方案 | 备选方案 |
|------|----------|----------|
| OpenClaw 飞书 | `feishu_ask_user_question`（交互式卡片） | 文本编号列表 |
| CLI / Hermes | 原生交互组件 | 文本编号列表 |
| 通用 | 文本编号列表 | — |

**feishu_ask_user_question 参数结构：**

```javascript
{
  questions: [
    {
      question: "问题文本（支持 lark_md）",
      header: "短标签（最多12字符）",
      options: [
        { label: "选项A", description: "补充说明" }
      ],
      multiSelect: false
    }
  ]
}
```

- 最多 4 个问题
- `options` 为空数组 → 文本输入框
- `options` 有内容 → 下拉选择器

## 工作流程

### 1. 执行启动（交互确认）

**步骤 1a — 展示执行计划摘要（Agent 主动）**

Agent 自动生成并发送计划摘要，包含：
- 任务清单（名称、负责人、截止日期）
- 关键依赖关系
- 资源分配概览
- 里程碑时间节点

不需要用户输入，纯信息推送。

**步骤 1b — 确认资源到位情况（交互）**

Agent 预判：大部分情况下资源"全部到位"，但需要确认。

飞书卡片示例：

```javascript
{
  questions: [{
    question: "执行计划如上，请确认资源到位情况：",
    header: "资源确认",
    options: [
      { label: "全部到位，可以启动", description: "人、物、工具均已就绪" },
      { label: "部分到位，需补充", description: "缺部分资源，需说明" },
      { label: "存在风险", description: "有关键资源未落实" }
    ],
    multiSelect: false
  }]
}
```

Fallback 文本格式：

```
请确认资源到位情况：
1. 全部到位，可以启动
2. 部分到位，需补充
3. 存在风险
0. 自定义输入
```

**如果用户选择"部分到位"→** Agent 追加文本输入问题让用户说明缺什么。

**如果用户选择"存在风险"→** 进入步骤 1c。

**步骤 1c — 阻塞项处理（条件交互，仅当存在阻塞时触发）**

飞书卡片示例：

```javascript
{
  questions: [{
    question: "检测到以下阻塞项：{阻塞项列表}\n请选择处理方式：",
    header: "阻塞处理",
    options: [
      { label: "延期启动", description: "等阻塞解决后再开始" },
      { label: "替代方案", description: "用其他资源/方式替代" },
      { label: "先启动可执行部分", description: "非阻塞任务先动起来" }
    ],
    multiSelect: false
  }]
}
```

Fallback 文本格式：

```
检测到以下阻塞项：
- {阻塞项1}
- {阻塞项2}

请选择处理方式：
1. 延期启动
2. 替代方案
3. 先启动可执行部分
0. 自定义输入
```

### 2. 执行监控（Agent 主动巡检 + 按需交互）

Agent 通过 cron 驱动定期巡检，**不主动打扰用户**。仅在发现以下异常时触发交互：

#### 异常类型一：进度偏差

当实际进度与计划偏差超过阈值时，Agent 先分析可能原因并排序，再发送交互。

飞书卡片示例：

```javascript
{
  questions: [
    {
      question: "任务「{任务名}」进度偏差：计划 {计划进度}%，实际 {实际进度}%。\nAgent 分析可能原因如下，请确认或补充：",
      header: "偏差原因",
      options: [
        { label: "资源不足", description: "人力/物料/预算不到位" },
        { label: "需求变更", description: "执行过程中需求发生调整" },
        { label: "技术障碍", description: "遇到未预期的技术问题" },
        { label: "人员变动", description: "关键人员请假/离职/调岗" }
      ],
      multiSelect: true
    },
    {
      question: "请补充具体原因（可选）：",
      header: "补充说明",
      options: [],
      multiSelect: false
    }
  ]
}
```

Fallback 文本格式：

```
⚠️ 任务「{任务名}」进度异常
计划进度：{计划进度}% → 实际进度：{实际进度}%

可能原因（可多选）：
1. 资源不足
2. 需求变更
3. 技术障碍
4. 人员变动
0. 自定义输入
```

#### 异常类型二：里程碑延期

飞书卡片示例：

```javascript
{
  questions: [{
    question: "里程碑「{里程碑名}」预计延期 {N} 天（原定 {日期}）。\n请选择应对策略：",
    header: "延期应对",
    options: [
      { label: "追加资源", description: "投入更多人/预算追赶进度" },
      { label: "调整范围", description: "缩减非必要交付内容" },
      { label: "接受延期", description: "重新协商截止日期" },
      { label: "取消里程碑", description: "该里程碑不再需要" }
    ],
    multiSelect: false
  }]
}
```

Fallback 文本格式：

```
⚠️ 里程碑「{里程碑名}」预计延期 {N} 天
原定日期：{日期}

请选择应对策略：
1. 追加资源
2. 调整范围
3. 接受延期
4. 取消里程碑
0. 自定义输入
```

#### 异常类型三：质量问题

飞书卡片示例：

```javascript
{
  questions: [{
    question: "检测到质量异常：{具体问题描述}\n请选择处理方式：",
    header: "质量处理",
    options: [
      { label: "暂停整改", description: "停下当前工作，优先解决质量问题" },
      { label: "继续执行并记录", description: "不中断流程，记录问题后续处理" },
      { label: "返工", description: "退回重做不合格部分" }
    ],
    multiSelect: false
  }]
}
```

### 3. 异常处理（交互确认）

**处理流程：**

1. Agent 自动分析异常原因（从执行日志、进度数据中提取）
2. Agent 预判 2-3 个可行处理方案，评估每个方案的影响
3. 发送交互：异常摘要 + 方案选项
4. 用户选择后，Agent 更新执行计划并记录决策

**交互规则：**
- 方案选项必须包含预期影响说明（时间、资源、质量）
- "自定义方案"始终作为兜底
- Agent 需要根据用户选择计算后续计划调整，并发送确认

飞书卡片示例：

```javascript
{
  questions: [
    {
      question: "**异常摘要：** {异常描述}\n**影响范围：** {受影响任务/里程碑}\n\n**方案 A**：{描述}（预计增加 {N} 天，需 {资源}）\n**方案 B**：{描述}（预计削减 {范围}，按期完成）\n**方案 C**：{描述}（预计延期 {N} 天）",
      header: "异常决策",
      options: [
        { label: "方案 A", description: "增加资源追赶进度" },
        { label: "方案 B", description: "缩减范围按期交付" },
        { label: "方案 C", description: "接受延期" }
      ],
      multiSelect: false
    },
    {
      question: "其他补充说明或自定义方案（可选）：",
      header: "补充",
      options: [],
      multiSelect: false
    }
  ]
}
```

---

### 4. 过程记录（Agent 自动，无需用户交互）

以下工作由 Agent 自动完成，不打扰用户：

- 自动收集关键指标数据
- 更新执行日志（`Do阶段/执行日志.md`）
- 记录异常处理过程和决策
- 同步进度到进度报告（`Do阶段/进度报告.md`）

---

## 监控指标

| 维度 | 指标 |
|------|------|
| 进度 | 计划完成率、里程碑达成率、时间偏差率 |
| 质量 | 一次成功率、缺陷密度、返工率 |
| 资源 | 资源利用率、效率指数、成本偏差 |

Agent 持续采集上述指标，当任一指标偏离基准超过阈值时，按异常处理流程触发交互。

---

## 时间预警

| 级别 | 条件 | 行动 |
|------|------|------|
| 🟡 黄色 | 超过计划 80% | Agent 记录日志，巡检频率提高 |
| 🟠 橙色 | 超过计划 100% | Agent 主动推送预警 + 触发偏差原因交互 |
| 🔴 红色 | 超过计划 120% | 必须决策，触发里程碑延期交互 |

---

## 完成检查

Agent 在 Do 阶段结束时自动检查以下项，全部通过才进入 Check 阶段：

- [ ] 计划任务全部完成
- [ ] 过程数据完整记录
- [ ] 执行问题已处理
- [ ] 质量标准已验证

如果存在未完成项，Agent 发送摘要并询问是否进入 Check 阶段或继续执行。

---

## 输出文件

| 文件 | 说明 |
|------|------|
| `Do阶段/执行日志.md` | 过程记录、异常日志、决策记录 |
| `Do阶段/进度报告.md` | 进度数据、指标趋势、里程碑状态 |
| `Do阶段/问题记录.md` | 问题清单、根因分析、处理结果 |

---

## 交互克制原则

> Do 阶段的核心是"做"，不是"聊"。

- Agent 80% 的时间自主工作（巡检、记录、分析）
- 只在**异常**或**关键决策**时才发起交互
- 每次交互都必须带来决策价值，不问"废话问题"
- 用户回复后 Agent 立即执行，不追问
