# 任务自动生成器

问题分析完成后，根据选择的 MECE 框架自动生成任务列表，写入 Bitable 任务表并同步创建飞书 Task。

---

## 概述

**用途**：在 Plan 阶段问题分析完成后，根据选择的 MECE 框架自动生成结构化的任务列表。

**核心价值**：
- 确保任务覆盖所有分析维度，避免遗漏
- 自动化任务创建流程，减少人工操作
- 统一任务管理入口（Bitable + 飞书 Task）

---

## 触发时机

满足以下条件时触发任务自动生成：

1. **Plan 阶段问题分析已完成**
   - 问题描述已记录
   - 根本原因分析完成（5Why 验证通过）
   - MECE 框架已选择

2. **MECE 框架已确认**
   - 从 11 个框架中选择了一个
   - 用户已确认框架选择

3. **问题要素已提取**
   - 症状（Symptoms）：问题的具体表现
   - 影响（Impact）：问题造成的损失或影响范围
   - 历史（History）：问题发生的时间线和背景

---

## 生成流程

```
问题分析完成
    ↓
AI 提取问题要素（症状、影响、历史）
    ↓
根据选择的 MECE 框架，按维度拆解
    ↓
每个维度生成 1-N 个具体任务
    ↓
写入 Bitable 任务表（批量创建）
    ↓
同步创建飞书 Task（串行调用）
    ↓
关联飞书 Task ID 回写到 Bitable
```

## 任务生成规则

### 数量规则

- **每个维度生成 1-3 个任务**
- 优先级高的维度生成 3 个
- 优先级中的维度生成 2 个
- 优先级低的维度生成 1 个

### 优先级判定

| 优先级 | 判定条件 | 说明 |
|--------|---------|------|
| 高 | 对问题影响程度 > 40% | 根因分析中影响最大的维度 |
| 中 | 对问题影响程度 20%-40% | 中等影响的维度 |
| 低 | 对问题影响程度 < 20% | 影响较小或预防性的维度 |

### 依赖关系识别

- **明确前置任务**：某些任务必须在其他任务完成后才能开始
- **标注依赖类型**：
  - `FS` (Finish-to-Start)：前置任务完成后，后续任务才能开始
  - `SS` (Start-to-Start)：前置任务开始后，后续任务才能开始
  - `FF` (Finish-to-Finish)：前置任务完成后，后续任务才能完成

### 时间分配规则

- **按项目周期比例分配**：
  - Plan 阶段任务：占项目总周期的 20%
  - Do 阶段任务：占项目总周期的 50%
  - Check 阶段任务：占项目总周期的 15%
  - Act 阶段任务：占项目总周期的 15%

- **截止日期计算**：
  - 任务截止日期 = 项目开始日期 + (项目周期 × 阶段权重 × 任务序号/该阶段任务数)

## API 调用流程

### 步骤 1：写入 Bitable 任务表

使用 `feishu_bitable_app_table_record.create` 批量创建任务记录。

```json
{
  "app_token": "<app_token>",
  "table_id": "<tasks_table_id>",
  "fields": {
    "任务ID": "uuid-xxx",
    "项目ID": "<project_id>",
    "任务标题": "操作技能培训 - 车床操作规范",
    "来源维度": "Man（人）",
    "来源类型": "问题分析生成",
    "任务描述": "针对车床操作不规范问题，开展全员SOP培训和考核",
    "负责人": [{"id": "ou_xxx"}],
    "截止日期": 1715308800000,
    "优先级": "高",
    "状态": "未开始"
  }
}
```

**重要坑位**：
- 人员字段必须是 `[{id: "ou_xxx"}]`，不能传字符串
- 日期字段使用**毫秒时间戳**
- 同一表不支持并发写，串行调用并延迟 0.5-1 秒

### 步骤 2：创建飞书 Task

使用 `feishu_task_task.create` 创建飞书任务。

```json
{
  "summary": "[PDCA-挖掘机臂架] T001 操作技能培训",
  "due": 1715308800000,
  "assignee": "ou_xxx",
  "description": "针对车床操作不规范问题，开展全员SOP培训和考核\n\n来源：问题分析生成\n维度：Man（人）\n优先级：高"
}
```

### 步骤 3：关联飞书 Task ID

创建飞书 Task 成功后，获取返回的 `task_id`，更新 Bitable 记录。

```json
{
  "record_id": "<bitable_record_id>",
  "fields": {
    "飞书任务ID": "<feishu_task_id>"
  }
}
```

## 任务命名规范

### 格式

```
[项目简称] T[序号] 任务标题
```

### 示例

- `[PDCA-挖掘机] T001 操作技能培训`
- `[PDCA-OEE提升] T005 设备维护计划制定`
- `[PDCA-焊接质量] T012 焊接工艺优化`

### 编号规则

- 序号从 001 开始，按创建顺序递增
- 同一维度内任务编号连续
- 不同维度间编号可以跳跃（预留调整空间）

## 错误处理

### Bitable 创建失败

1. 检查 `app_token` 和 `table_id` 是否正确
2. 检查字段格式是否符合要求（人员、日期、单选等）
3. 检查是否有并发写入问题
4. 如果失败，停止流程并报告错误，不创建飞书 Task

### 飞书 Task 创建失败

1. 记录失败的任务 ID
2. 继续创建其他任务
3. 完成后汇总失败列表，提示用户手动创建
4. Bitable 中 `飞书任务ID` 字段留空，标注为待手动创建
